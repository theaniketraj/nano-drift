# Project Structure — Nano Drift

**v0.1.0** · npm workspaces monorepo · TypeScript throughout

---

## 1. Architecture Overview

Nano Drift is split into two runtime processes that communicate over a local
WebSocket connection.

```bash
┌──────────────────────────────────────────────────────────────┐
│  VS Code Extension Host (Node.js, same process as VS Code)   │
│                                                              │
│  extension.ts ──registers──▶ commands/*                      │
│                              │                               │
│  DaemonClient ◀──────────────┘  (shared dep via CommandDeps)│ 
│       │  WebSocket ws://127.0.0.1:27183/rpc                  │
│       │  binary stream  ws://127.0.0.1:27183/screen          │
└───────┼──────────────────────────────────────────────────────┘
        │  (spawned by DaemonClient if not already running)
┌───────▼──────────────────────────────────────────────────────┐
│  Nano Drift Daemon (child Node.js process)                   │
│                                                              │
│  DaemonServer                                                │
│   ├── /rpc    JSON-RPC 2.0-style request/response + push     │
│   ├── /screen binary PNG frame broadcast                     │
│   ├── AdbManager    ──execFile──▶ adb                        │
│   ├── GradleRunner  ──spawn────▶ ./gradlew                   │
│   ├── ScreenStreamer ──execFile──▶ adb exec-out screencap    │
│   └── FileWatcher   ──chokidar──▶ app/src/**                 │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│  GitHub Pages (static, no backend)                           │
│  landing/   ──served at── theaniketraj.github.io/nano-drift  │
└──────────────────────────────────────────────────────────────┘
```

**Key design decisions:**

- **Separate process for ADB/Gradle.** The extension host is single-threaded
  and must remain responsive. Long-running operations (builds can take minutes)
  run in the daemon and communicate results via push events.
- **Loopback-only WebSocket.** The daemon binds `127.0.0.1` exclusively.
  There is no configuration to widen the bind address.
- **No shell string construction.** Every `adb` and `gradlew` invocation uses
  `execFile` or `spawn` with an explicit string array — never a shell-expanded
  command string.
- **Single active connection.** The server supports multiple concurrent
  `/rpc` clients (tracked in `rpcClients: Set<WebSocket>`) to broadcast push
  events, but in practice a single extension instance connects. The `/screen`
  endpoint supports multiple simultaneous WebView clients.

### 3.1 Activation & lifecycle (`extension.ts`)

`activate()` is called by VS Code when any of the `activationEvents` fire:

```json
"activationEvents": [
  "*",
  "workspaceContains:**/AndroidManifest.xml",
  "workspaceContains:**/build.gradle",
  "workspaceContains:**/build.gradle.kts"
]
```

Activation sequence:

1. **SDK detection** — `detectAndroidSdk()` is called; if it returns
   `undefined` a warning message is shown with a link to open settings.
2. **Manager instantiation** — `StatusBarManager`, `DaemonClient`, and
   `DiagnosticsManager` are created and pushed to `context.subscriptions`
   (VS Code disposes them on deactivation).
3. **Build event wiring** — `daemonClient.onBuildProgress` events drive the
   status bar states (`setBuilding → setDeploying → setRunning / setError`)
   and route compiler errors to `DiagnosticsManager.update()`.
4. **Command registration** — `registerCommands()` is called; this also
   creates the `DeviceScreenViewProvider` and registers it with
   `vscode.window.registerWebviewViewProvider`.
5. **Device restoration** — the last-used device serial is persisted in
   `context.workspaceState` under key `nanoDrift.lastDevice`. On activation
   the daemon is queried for the current device list and, if the saved serial
   is present, it is re-selected automatically. A `onDeviceListChanged`
   subscription handles the race where the daemon isn't up yet at activation
   time.
6. **File watcher auto-start** — if `nanoDrift.autoRunOnSave` is `true` and
   there are open workspace folders, `daemonClient.startWatcher()` is called
   for each.

### 3.2 Daemon client (`daemon/client.ts`)

`DaemonClient` is the single object the extension uses to talk to the daemon.
It owns the WebSocket connection and the daemon child process lifetime.

**Connection lifecycle:**

```bash
ensureRunning()
  ├── ws?.readyState === OPEN  →  return immediately
  ├── connect() succeeds       →  ws assigned, message handler attached
  └── connect() fails
        ├── spawnDaemon()      →  node daemon/out/index.js --port <N>
        └── retryConnect()     →  12 attempts × 500 ms back-off
```

`spawnDaemon()` resolves the daemon entry point at
`__dirname/../../../daemon/out/index.js` (relative to the compiled `out/`
directory) and spawns it with `stdio: ['ignore', 'pipe', 'pipe']`. stdout
and stderr are piped to the Nano Drift `OutputChannel`.

**Message routing:**

Every incoming WebSocket frame is parsed as JSON. If a frame has no `id`
field but has a `method` field it is a **push notification** from the daemon
and is dispatched to `handlePush()`. Frames with an `id` are responses to
outgoing RPC calls and are resolved by per-call `message` listeners attached
inline inside `rpc<T>()`.

**Push events emitted as VS Code event emitters:**

| Push method       | VS Code event                      | Consumers                             |
| ----------------- | ---------------------------------- | ------------------------------------- |
| `build.progress`  | `DaemonClient.onBuildProgress`     | `extension.ts`, output channel mirror |
| `devices.changed` | `DaemonClient.onDeviceListChanged` | `extension.ts` (device restoration)   |

**Public API surface:**

| Method                                   | RPC method called      |
| ---------------------------------------- | ---------------------- |
| `listDevices()`                          | `devices.list`         |
| `listAvds()`                             | `emulator.listAvds`    |
| `startEmulator(avdName)`                 | `emulator.start`       |
| `connectWifi(address)`                   | `adb.connectWifi`      |
| `pairDevice(address, code)`              | `adb.pair`             |
| `waitForBoot(knownSerials)`              | `emulator.waitForBoot` |
| `setActiveDevice(serial)`                | `devices.setActive`    |
| `detectPackage(projectPath)`             | `adb.detectPackage`    |
| `startWatcher(projectPath, packageName)` | `watcher.start`        |
| `stopWatcher(projectPath?)`              | `watcher.stop`         |
| `build(projectPath, packageName?)`       | `gradle.build`         |
| `deploy()`                               | `adb.launch`           |
| `sendTap(x, y)`                          | `adb.tap`              |
| `sendSwipe(x1, y1, x2, y2)`              | `adb.swipe`            |
| `sendText(text)`                         | `adb.sendText`         |
| `sendKey(keycode)`                       | `adb.sendKey`          |
| `getScreenSize(serial)`                  | `screen.getSize`       |

**Gradle arg resolution** (`resolveGradleArgs()`):
The method derives the Gradle task from user configuration before any
`build` or `watcher.start` call:

```bash
nanoDrift.gradleArgs contains an explicit task (non-flag entry)?
  YES → use gradleArgs as-is
  NO  → derive task from nanoDrift.buildVariant
          'debug'   → installDebug
          'release' → installRelease
          <x>       → install<X>
        then append any --flag entries from gradleArgs
```

### 3.3 Commands

All commands share a `CommandDeps` object (defined in `commands/index.ts`)
that holds references to the four singletons created in `extension.ts`:

```typescript
interface CommandDeps {
  context: vscode.ExtensionContext;
  statusBarManager: StatusBarManager;
  daemonClient: DaemonClient;
  sdkPath: string | undefined;
  diagnosticsManager: DiagnosticsManager;
}
```

This avoids global state — commands receive everything they need as a
parameter rather than importing module-level singletons.

| Command ID                   | Handler file          | Description                                                                                 |
| ---------------------------- | --------------------- | ------------------------------------------------------------------------------------------- |
| `nanoDrift.runOnTheFly`      | `runOnTheFly.ts`      | Resolves package name, calls `daemonClient.build()`, updates status bar and diagnostics     |
| `nanoDrift.selectDevice`     | `selectDevice.ts`     | QuickPick populated from `daemonClient.listDevices()`; persists selection to workspaceState |
| `nanoDrift.startEmulator`    | `startEmulator.ts`    | Lists AVDs, prompts user, calls `daemonClient.startEmulator()`, polls `waitForBoot()`       |
| `nanoDrift.connectWifi`      | `connectWifi.ts`      | Input box for IP:port or pairing code, calls `connectWifi()` / `pairDevice()`               |
| `nanoDrift.showDeviceScreen` | `showDeviceScreen.ts` | Focuses the sidebar view (`nanoDrift.deviceScreenView.focus`)                               |
| `nanoDrift.listDevices`      | `commands/index.ts`   | Inline — shows device list in an information message                                        |
| `nanoDrift.stopDaemon`       | `commands/index.ts`   | Inline — calls `daemonClient.stop()`                                                        |

**Package name resolution** (inside `runOnTheFly.ts`):

```bash
1. nanoDrift.packageName setting (if non-empty)         ← user override
2. daemonClient.detectPackage(workspaceFolder)          ← parse AndroidManifest.xml
3. error shown, command aborted                         ← neither available
```

### 3.4 WebviewViewProvider — device screen (`showDeviceScreen.ts`)

`DeviceScreenViewProvider` implements `vscode.WebviewViewProvider` and is
registered against view ID `nanoDrift.deviceScreenView`.

The view is declared in `package.json` as a sidebar webview inside the
`nano-drift` activity-bar container:

```json
"views": {
  "nano-drift": [{ "type": "webview", "id": "nanoDrift.deviceScreenView", "name": "Device Screen" }]
}
```

The option `retainContextWhenHidden: true` is passed on registration so the
WebView and its WebSocket connection survive when the user switches to a
different sidebar panel.

**`resolveWebviewView()` flow:**

1. `webview.options = { enableScripts: true }` — scripts allowed, no local
   resource roots (all assets are inline).
2. `buildWebviewHtml(daemonPort)` — returns a self-contained HTML string with
   inline CSS and JS. The daemon port is interpolated into the CSP header
   (`connect-src ws://localhost:<port>`) and the JS WebSocket URL.
3. If an active device is known, `daemonClient.getScreenSize()` is called and
   the result is posted as `{ type: 'screenSize', width, height }` before
   any frames arrive so the canvas pre-sizes correctly.
4. `onDidReceiveMessage` handles input events from the WebView:

| Message type   | Fields                 | Action                                                         |
| -------------- | ---------------------- | -------------------------------------------------------------- |
| `tap`          | `x`, `y`               | `daemonClient.sendTap(x, y)`                                   |
| `swipe`        | `x1`, `y1`, `x2`, `y2` | `daemonClient.sendSwipe(...)`                                  |
| `key`          | `keycode`              | `daemonClient.sendKey(keycode)`                                |
| `text`         | `text`                 | `daemonClient.sendText(text)`                                  |
| `openUrl`      | `url`                  | `vscode.env.openExternal(Uri.parse(url))`                      |
| `openSettings` | —                      | `executeCommand('workbench.action.openSettings', 'nanoDrift')` |
| `runOnTheFly`  | —                      | `executeCommand('nanoDrift.runOnTheFly')`                      |

**WebView Content Security Policy:**

```bash
default-src 'none';
script-src  'unsafe-inline';
style-src   'unsafe-inline';
connect-src ws://localhost:<daemonPort>;
```

Scripts and styles are self-contained inline strings; external network access
is restricted to the daemon WebSocket only.

**Screen frame rendering in the WebView:**
The WebView opens a second WebSocket to `ws://localhost:<port>/screen`.
Incoming binary frames (raw PNG bytes) are written to an `<img>` tag via
`URL.createObjectURL(new Blob([frame], { type: 'image/png' }))`. The
previous object URL is revoked immediately to avoid memory leaks. The
phone-bezel UI (CSS `#phone-outer`) adapts aspect ratio from the first frame
dimensions.

### 3.5 Supporting modules

**`sdk.ts`** — `detectAndroidSdk(): Promise<string | undefined>`

Resolution order:

1. `nanoDrift.androidHome` VS Code setting (if non-empty and directory exists)
2. `ANDROID_HOME` environment variable
3. `ANDROID_SDK_ROOT` environment variable
4. Returns `undefined` — SDK not found

Also exports `resolveAdb(sdkPath)` and `resolveEmulator(sdkPath)` which
append `platform-tools/adb[.exe]` and `emulator/emulator[.exe]` respectively.

---

**`statusBar.ts`** — `StatusBarManager`

Two `StatusBarItem` instances, both left-aligned:

| Item         | Priority | Command                  | States                                                        |
| ------------ | -------- | ------------------------ | ------------------------------------------------------------- |
| `deviceItem` | 100      | `nanoDrift.selectDevice` | `$(vm) <name>` / `$(device-mobile) <name>` / No Device        |
| `actionItem` | 99       | `nanoDrift.runOnTheFly`  | idle / `$(sync~spin) Building…` / deploying / running / error |

The `error` state sets `backgroundColor` to `statusBarItem.errorBackground`.

---

**`diagnostics.ts`** — `DiagnosticsManager`

Wraps a `vscode.DiagnosticCollection('nano-drift')`. On each build result
it calls `collection.clear()` then groups incoming `BuildError[]` by file
URI and calls `collection.set()` once per file.

Line/column translation: Gradle reports 1-based lines and columns; VS Code
`Range` constructor expects 0-based — `DiagnosticsManager` subtracts 1 from
each, clamped to ≥ 0.

File path resolution order:

1. Absolute path (or Windows drive-letter path) → `vscode.Uri.file()`
2. Relative path → resolved against `workspaceFolders[0]`

### 4.1 Entry point & server

**`index.ts`** uses `commander` to parse a single `--port` flag (default
`27183`). It instantiates `DaemonServer`, calls `server.start()`, and
registers `SIGINT` / `SIGTERM` handlers that call `server.stop()` before
`process.exit(0)`.

**`DaemonServer` (`server.ts`):**

The server creates a `http.Server` and attaches a `WebSocketServer` to it
(the `ws` package). All WebSocket routing is done by inspecting `req.url` in
the `'connection'` handler:

```bash
req.url.startsWith('/screen')  →  screen.addClient(ws)
otherwise                       →  rpcClients.add(ws); handleRpcClient(ws)
```

**RPC dispatch:**

```typescript
private handleRpcClient(ws: WebSocket): void {
    ws.on('message', (raw) => {
        const { id, method, params } = JSON.parse(raw);
        const handler = this.handlers.get(method);
        handler(params)
            .then(result => ws.send(JSON.stringify({ id, result })))
            .catch(err  => ws.send(JSON.stringify({ id, error: err.message })));
    });
}
```

Handlers are registered with a private `reg(method, fn)` helper inside
`registerHandlers()`. All handler functions are typed as
`(params: unknown) => Promise<unknown>` and cast params internally.

**Push broadcast (`push(method, params)`):**
Iterates `rpcClients` and sends to all `OPEN` connections:

```typescript
push(method: string, params: unknown): void {
    const frame = JSON.stringify({ method, params });
    for (const client of this.rpcClients) {
        if (client.readyState === WebSocket.OPEN) client.send(frame);
    }
}
```

**Device polling:**
`startDevicePolling()` sets a 3-second interval that calls
`adb.listDevices()`, compares serialized device serial arrays, and fires
`push('devices.changed', devices)` only when the list actually changes.
This avoids flooding the extension with no-op events.

**Build-deploy cycle:**
`scheduleAutoBuild(opts)` debounces file change events with a 300 ms timer.
`runBuildCycle(opts)` is the actual execution:

```bash
push build.progress { stage: 'building' }
gradle.build(projectPath, gradleArgs, onLine)
  └── onLine → push build.progress { stage: 'output', line }
push build.progress { stage: 'deploying' }
adb.launch(serial, packageName)
push build.progress { stage: 'done', errors }
-- or --
push build.progress { stage: 'error', message, errors }
```

A per-project `buildInProgress` flag prevents concurrent builds for the same
project path.

**Registered RPC handlers:**

| Method                 | Params                                      | Delegates to                      |
| ---------------------- | ------------------------------------------- | --------------------------------- |
| `devices.list`         | —                                           | `adb.listDevices()`               |
| `devices.setActive`    | `{ serial }`                                | `adb.setActiveDevice()`           |

| `adb.connectWifi`      | `{ address }`                               | `adb.connectWifi()`               |
| `adb.pair`             | `{ address, code }`                         | `adb.pairDevice()`                |
| `adb.tap`              | `{ serial, x, y }`                          | `adb.tap()`                       |
| `adb.swipe`            | `{ serial, x1, y1, x2, y2 }`                | `adb.swipe()`                     |

### 4.2 ADB manager (`adb/index.ts`)

`AdbManager` wraps the `adb` binary using `child_process.execFile` (via
`util.promisify`) and `spawn`.

**Binary resolution:**

```typescript
const sdk =
  process.env["ANDROID_HOME"] ?? process.env["ANDROID_SDK_ROOT"] ?? "";
this.adbPath = sdk
  ? path.join(sdk, "platform-tools", isWin ? "adb.exe" : "adb")
  : "adb"; // fall back to PATH
```

**Key methods:**

| Method                              | ADB invocation                                                                                                            |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `listDevices()`                     | `adb devices -l` — parses `model:` token for friendly name                                                                |
| `startEmulator(avdName)`            | `emulator -avd <name> -no-window -no-audio -no-boot-anim` (detached, unref'd)                                             |
| `connectWifi(address)`              | `adb connect <address>`                                                                                                   |
| `pairDevice(address, code)`         | `adb pair <address> <code>` (Android 11+ wireless)                                                                        |
| `waitForEmulatorBoot(knownSerials)` | Polls `adb devices` for new emulator serial, then polls `adb shell getprop sys.boot_completed` until `"1"`, timeout 120 s |
| `tap(serial, x, y)`                 | `adb -s <serial> shell input tap <x> <y>`                                                                                 |
| `swipe(serial, x1, y1, x2, y2)`     | `adb -s <serial> shell input swipe <x1> <y1> <x2> <y2>`                                                                   |
| `sendKey(serial, keycode)`          | `adb -s <serial> shell input keyevent <keycode>`                                                                          |
| `sendText(serial, text)`            | `adb -s <serial> shell input text <escaped>`                                                                              |
| `launch(serial, packageName?)`      | `adb -s <serial> shell monkey -p <pkg> -c android.intent.category.LAUNCHER 1`                                             |
| `detectPackage(projectPath)`        | Reads `<projectPath>/app/src/main/AndroidManifest.xml`, extracts `package` attribute                                      |
| `getScreenSize(serial)`             | `adb -s <serial> shell wm size` — parses `Physical size: WxH`                                                             |

### 4.3 Gradle runner (`gradle/index.ts`)

`GradleRunner.build(projectPath, args, onLine?)` returns
`Promise<BuildError[]>`.

**Gradlew resolution:**

```typescript
private resolveGradlew(projectPath: string): string {
    const wrapper = path.join(projectPath, isWindows ? 'gradlew.bat' : 'gradlew');
    if (!fs.existsSync(wrapper)) throw new Error(...);
    return wrapper;
}
```

**Subprocess options:**

```typescript
spawn(gradlew, args, {
  cwd: projectPath,
  stdio: "pipe",
  shell: process.platform === "win32", // required for .bat execution
});
```

`shell: true` is used on Windows only because `.bat` files require it. On
macOS/Linux `spawn` is called without a shell.

**Output processing:**
stdout and stderr are both processed through the same `processLine()` function,
which calls the optional `onLine` callback (for push streaming) and
`parseGradleLine()` (for error extraction).

**`parseGradleLine(line)`** handles three compiler output formats:

| Format       | Pattern                              | Example                                          |
| ------------ | ------------------------------------ | ------------------------------------------------ |
| Kotlin (new) | `e: file:///path.kt:10:5: message`   | `e: file:///app/src/Main.kt:42:13: unresolved`   |
| Kotlin (old) | `e: /path.kt: (10, 5): message`      | `e: /app/src/Main.kt: (42, 13): unresolved`      |
| Java         | `/path/File.java:10: error: message` | `/app/src/Main.java:42: error: cannot find sym.` |

Returns a `BuildError | undefined` — only lines that match a known pattern
produce a diagnostic entry.

### 4.4 Screen streamer (`screen/index.ts`)

`ScreenStreamer` manages all `/screen` WebSocket clients and runs a
self-scheduling capture loop.

**Adaptive frame rate:**

```bash
MIN_INTERVAL_MS = 67   (≈ 15 fps ceiling)

after each capture:
  delay = max(0, 67 - captureMs)
  setTimeout(captureFrame, delay)
```

If the device is slow to respond the next capture is scheduled with zero
delay (runs immediately after the previous one completes), naturally reducing
fps under load without any explicit back-pressure mechanism.

**Screencap invocation:**

```typescript
execFile(adbBin, ["-s", serial, "exec-out", "screencap", "-p"], {
  encoding: "buffer",
  maxBuffer: 16 * 1024 * 1024, // 16 MiB limit
});
```

The PNG bytes are broadcast as binary WebSocket frames directly — no
transcoding. The WebView decodes them using the browser's native PNG decoder
via `Blob` + `URL.createObjectURL`.

**Client lifecycle:**

- First client added → `startCapture()` starts the loop.
- Last client removed → `stopCapture()` clears the pending timer.

### 4.5 File watcher (`watcher/index.ts`)

`FileWatcher` wraps `chokidar.watch()`. It watches:

```bash
<projectPath>/app/src/**
<projectPath>/app/build.gradle[.kts]
<projectPath>/build.gradle[.kts]
<projectPath>/settings.gradle[.kts]
```

Ignored patterns:

- `/(^|[/\\])\../` — dotfiles and hidden directories
- `/[/\\]build[/\\]/` — Gradle build output directories
- `/[/\\]\.gradle[/\\]/` — Gradle caches

`awaitWriteFinish: { stabilityThreshold: 300, pollInterval: 100 }` prevents
false triggers from editors that write files in multiple chunks.

The 300 ms `stabilityThreshold` is complementary to the 300 ms debounce in
`DaemonServer.scheduleAutoBuild()` — together they ensure at most one build
fires per save, even for rapid multi-file saves.

---

## 5. RPC Protocol

The transport is a JSON-based request/response protocol over WebSocket.

**Request (extension → daemon):**

```jsonc
{
  "id": "abc123",
  "method": "gradle.build",
  "params": { "projectPath": "/..." },
}
```

**Response (daemon → extension):**

```jsonc
{ "id": "abc123", "result": [] }           // success
{ "id": "abc123", "error": "message" }     // failure
```

**Push notification (daemon → extension, no id):**

```jsonc
{ "method": "build.progress", "params": { "stage": "building" } }
{ "method": "devices.changed", "params": [{ "serial": "emulator-5554", ... }] }
```

Push notifications have no `id` field. The extension distinguishes them from
responses by checking `!msg.id && msg.method`.

The screen stream (`/screen` endpoint) is a separate WebSocket that carries
raw binary PNG frames with no framing metadata — each WebSocket message is
exactly one PNG image.

## 8. Code Conventions

### TypeScript

- **Strict mode enabled** — `"strict": true` in all `tsconfig.json` files.
  No `any` casts without a comment explaining why.
- **`void` on ignored promises** — fire-and-forget async calls are prefixed
  with `void` rather than left as floating promises. ESLint would flag
  unawaited promises otherwise.
- **No class-level `this` leaks** — event-handler callbacks always use
  `.bind(this)` or arrow method properties. `handleMessage.bind(this)` is
  passed to `ws.on('message', ...)`.
- **Type imports** — `import type { ... }` is used where only type
  information is needed (avoids circular runtime imports between `commands/`
  modules and `daemon/client.ts`).
- **`interface` for data shapes, `class` for stateful objects** — plain
  data transfer objects (`DeviceInfo`, `BuildError`, `RpcMessage`) are
  interfaces; everything with lifecycle (`DaemonClient`, `AdbManager`, etc.)
  is a class implementing `vscode.Disposable` where appropriate.

### Error handling

- Daemon RPC errors propagate as thrown `Error` objects (the daemon sends
  `{ id, error: "message" }` and the client rejects the promise).
- ADB command failures check stdout content for known failure strings
  (e.g. `connectWifi` checks for `"connected"`, `pairDevice` checks for
  `"successfully paired"`) rather than relying solely on exit codes.
- `child_process.execFile` calls set `timeout: 30_000` (30 s) to prevent
  indefinitely hanging processes from stalling the daemon.

### Naming

- Files use `camelCase` for modules (`statusBar.ts`, `runOnTheFly.ts`).
- Classes use `PascalCase` (`DaemonServer`, `AdbManager`).
- VS Code command IDs follow `nanoDrift.<camelCaseName>` (e.g.
  `nanoDrift.runOnTheFly`).
- RPC method names follow `<noun>.<verb>` (e.g. `devices.list`,
  `gradle.build`, `adb.tap`).
- Private class members that back a VS Code `EventEmitter` are named
  `_onXxx` (underscore prefix); the public `.event` getter is named `onXxx`.

### Comments

- Section dividers in longer files use:

  ```typescript
  // ------------------------------------------------------------------ //
  //  Section name
  // ------------------------------------------------------------------ //
  ```

- JSDoc comments are written only on public API surfaces and non-obvious
  algorithmic choices — not on trivial getters or self-evident one-liners.
